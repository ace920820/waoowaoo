# Phase 03.3 Research — Import API And Persistence

## Goal

Plan the backend import API for director-authored `waoo.storyboard_package` files after Phase 03.2 delivered the pure schema, parser, validator, and mapper contract.

## Boundary

- In scope: backend route, preview mode, commit mode, asset matching, shot-group/draft metadata persistence, integration tests.
- Out of scope: script-page upload UI, browser file handling, navigation, and real image/video generation after import.

## Candidate File Map

### Likely

- `src/app/api/novel-promotion/[projectId]/multi-shot-drafts/route.ts` — existing secured episode-level multi-shot draft materialization route; shows auth, episode ownership, shot group reuse, and draft metadata snapshot patterns.
- `src/app/api/novel-promotion/[projectId]/shot-groups/route.ts` — existing shot group update/video config route; confirms request/transaction conventions.
- `src/app/api/novel-promotion/[projectId]/assets/route.ts` — existing project asset read surface; confirms characters, locations, and props returned from project assets.
- `src/lib/novel-promotion/storyboard-package/*` — Phase 03.2 parser/validator/mapper to reuse directly.
- `src/lib/shot-group/draft-metadata.ts` — canonical draft metadata normalization and asset binding fields.
- `src/lib/shot-group/video-config-snapshot.ts` — canonical way to store video config plus draft metadata in `videoReferencesJson`.
- `prisma/schema.prisma` — confirms `NovelPromotionShotGroup`, `NovelPromotionShotGroupItem`, `NovelPromotionCharacter`, `NovelPromotionLocation`, and props-as-location-with-`assetKind=prop` persistence.
- `tests/integration/api/specific/novel-promotion-multi-shot-drafts-route.test.ts` — best integration-test fixture style for episode shot-group creation and metadata preservation.

### Maybe

- `src/lib/novel-promotion/multi-shot/persist-drafts.ts` — may provide reusable persistence patterns, but import should likely use a dedicated service to avoid coupling to generated draft rebuilding.
- `src/lib/media/attach.ts` — useful only if import route response needs media fields on returned shot groups.
- `tests/integration/api/specific/novel-promotion-shot-groups-route.test.ts` — useful for transaction and route mock patterns.

### Avoid For 03.3

- Script page workspace UI and hooks — belongs to Phase 03.4.
- Workers and generation handlers — import must prefill production data only, not trigger media generation.
- Prompt files — package content is already authored by director agents and parsed by 03.2.

## Current Architecture Findings

1. Novel-promotion API routes use `apiHandler`, `requireProjectAuthLight`, `isErrorResponse`, and ownership helpers such as `buildEpisodeInProjectWhere`.
2. Existing multi-shot shot groups persist:
   - group fields: `title`, `templateKey`, `groupPrompt`, `videoPrompt`, audio/dialogue/reference flags, `videoModel`, `videoReferencesJson`.
   - items: `NovelPromotionShotGroupItem` rows keyed by `shotGroupId + itemIndex`, with `title` and `prompt` fields.
3. Rich draft metadata lives inside `videoReferencesJson` via `buildShotGroupVideoConfigSnapshot` and `parseShotGroupDraftMetadata`.
4. `draft-metadata.ts` already models selected/preselected/script-derived/effective location, character, and prop bindings.
5. Project props are stored as `NovelPromotionLocation` rows where `assetKind === 'prop'`; locations are rows where `assetKind !== 'prop'`.
6. Existing multi-shot draft rebuild preserves saved editable metadata by parsing previous `videoReferencesJson`; the import route must avoid clobbering non-imported/manual groups and should define explicit overwrite scope.

## Recommended Design

Create a dedicated import service under `src/lib/novel-promotion/storyboard-package/import-service.ts` so the route stays thin and Phase 03.4 can call the same API contract.

Route shape:

- `POST /api/novel-promotion/[projectId]/episodes/[episodeId]/storyboard-package-import`
- Request body:
  - `mode: 'preview' | 'commit'`
  - `content: string`
  - optional `filename` / `contentType`
  - optional `overwriteStrategy`: default `replace-imported`
  - optional `preserveGeneratedMedia`: default `true`
- Preview returns parsed package summary, create/update plan, asset match results, warnings, validation issues, and no DB mutation.
- Commit repeats parse/preview in-process, then writes shot groups in a transaction.

Persistence identity:

- Store stable import identity in draft metadata:
  - `packageId`
  - `importedStoryboardPackageId`
  - `importedStoryboardSegmentId`
  - `segmentKey` from mapper, e.g. `${packageId}:${segmentId}`
- Detect existing imported shot groups by parsed draft metadata, scoped to same episode and package/segment identity.

Overwrite policy for 03.3:

- `replace-imported`: update only shot groups previously imported from the same package/segment identity; create missing groups; do not delete manual shot groups.
- Preserve generated media by default: keep `referenceImageUrl`, `compositeImageUrl`, `videoUrl`, media ids, and generated item `imageUrl` unless caller explicitly requests reset in a later phase.

Asset matching:

- Match package refs against declared package assets first via Phase 03.2 mapper output, then against project assets.
- Match by exact `externalId` against existing ids when possible, then case/trim-insensitive `matchName` or `name` against project asset names.
- For missing matches, keep script-derived fallback metadata and return warnings; never create assets silently.

## Test Strategy

Add a new focused integration test file:

- `tests/integration/api/specific/novel-promotion-storyboard-package-import-route.test.ts`

Test scenarios:

1. Preview parses fenced Markdown package, returns segments, shot counts, create plan, and asset matches without creating/updating shot groups.
2. Preview reports validation errors for invalid package content.
3. Commit creates shot groups/items and writes all required group fields and draft metadata.
4. Commit updates/replaces existing imported shot groups by package/segment identity.
5. Commit preserves generated media by default when replacing imported data.
6. Missing assets produce warnings and script-derived fallback metadata, not asset creation.
