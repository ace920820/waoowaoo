# Phase 03.1 Research: Multi-Shot Cinematic Prompting

## Current Architecture Findings

### LLM Entry

- `script-to-storyboard-stream` submits `TASK_TYPE.SCRIPT_TO_STORYBOARD_RUN`.
- `script-to-storyboard.ts` branches multi-shot episodes into `handleMultiShotScriptToStoryboardTask(...)`.
- The multi-shot handler calls the LLM once per coarse script clip with action `multi_shot_storyboard`.
- Prompt construction uses `PROMPT_IDS.NP_MULTI_SHOT_SEGMENTATION`.

### Prompt Template Location

- Prompt catalog entry: `src/lib/prompt-i18n/catalog.ts`
- Prompt ID: `src/lib/prompt-i18n/prompt-ids.ts`
- Chinese template: `lib/prompts/novel-promotion/multi_shot_segmentation.zh.txt`
- English template: `lib/prompts/novel-promotion/multi_shot_segmentation.en.txt`

### Current Output and Mapping

Current model output fields:

- `segmentIndexWithinClip`
- `title`
- `sceneLabel`
- `narrativePrompt`
- `embeddedDialogue`
- `shotRhythmGuidance`
- `expectedShotCount`

Current mapping:

- `narrativePrompt` -> `groupPrompt`
- `narrativePrompt` -> `videoPrompt`
- `narrativePrompt` -> review card fallback for `referencePromptText`
- default shot-group items receive titles only, not generated shot-level prompts

## Design Implications

1. A prompt-only change can improve prose quality but cannot fully solve downstream prompt specialization.
2. The existing data model already has a useful slot-level path through `NovelPromotionShotGroupItem.prompt`.
3. `draftMetadata` already supports `referencePromptText` and `compositePromptText`, so specialized prompt fields can be persisted without schema migration.
4. Backward compatibility should be implemented in the parser, not by asking downstream consumers to handle every old/new shape.
5. Downstream builders should be strengthened after persistence so tests can verify real prompt propagation.

## Recommended Implementation Shape

- Upgrade prompt templates to request richer JSON while keeping old required fields.
- Extend draft types and parser to accept new fields.
- Persist new fields into existing metadata and item prompt structures.
- Use shot-level prompts in composite and video prompt builders.
- Keep UI mostly unchanged in the first pass; seed existing fields from specialized values.

## Risks

- LLM may emit malformed nested JSON. Mitigate with tolerant parsing and fallback to old fields.
- Overly verbose `shots[]` may exceed model output reliability. Mitigate with compact schema and max 4-9 rows.
- UI may become confusing if too many fields surface. Mitigate by keeping detailed cinematic plan hidden initially.
- Regression risk in rebuild preservation. Mitigate with tests around previous metadata and user override preservation.
