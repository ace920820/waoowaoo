# Phase 03.4 Research — Script Page Upload UI

## Goal

Plan the script-page UI that lets users upload director-authored storyboard packages in multi-shot mode, preview the backend import plan, commit it, and enter the multi-shot confirmation / video workflow with imported fields prefilled.

## Boundary

- In scope: script-page upload button, `.md` / `.json` browser file read, preview modal/panel, commit action, route client mutation, query invalidation, stage navigation, focused UI tests.
- Out of scope: package schema/parser changes, backend import persistence changes, image/video generation, and real filesystem upload beyond browser `File.text()`.

## Current Architecture Findings

1. The script-page production mode card and primary “确认并开始绘制” button are rendered by `ScriptViewAssetsPanel.tsx`.
2. `ScriptViewRuntime.tsx` owns script-page local state and passes `onGenerateStoryboard` to `ScriptViewAssetsPanel`.
3. `useWorkspaceStageRuntime` exposes `onRunScriptToStoryboard`, `onStageChange`, and mode-specific state to `ScriptStage`.
4. `useNovelPromotionWorkspaceController` currently creates the `useEnsureEpisodeMultiShotDrafts` mutation and passes runtime props; it also has access to `onRefresh` and `handleStageChange`.
5. Existing React Query mutation patterns use `requestJsonWithError` and invalidate `queryKeys.episodeData(projectId, episodeId)` on settle.
6. Existing script-view tests render `ScriptViewAssetsPanel` directly with fake translators, so new UI should be testable without full workspace mounting.
7. Phase 03.3 API supports the route shape needed by the UI:
   - `POST /api/novel-promotion/[projectId]/episodes/[episodeId]/storyboard-package-import`
   - `mode: 'preview' | 'commit'`
   - `content`, `filename`, `contentType`, `overwriteStrategy`, `preserveGeneratedMedia`

## Recommended UI Design

1. Place a secondary “上传分镜表” button beside the primary “确认并开始绘制” button, visible only for `episodeProductionMode === 'multi_shot'`.
2. Use a hidden file input with `.md,.json,application/json,text/markdown,text/plain` accept types and `File.text()` for content.
3. After file selection, call preview API and show a modal/dialog with:
   - package title/language/scene count/segment count
   - segment rows: scene label, title, duration, template, shot count
   - prompt summaries: reference prompt and video prompt snippets when provided by preview or a lightweight client-derived summary
   - asset matching grouped by location/character/prop with matched vs script-derived fallback badges
   - overwrite strategy: create/update counts, preserve generated media default
   - validation/errors/warnings
4. “确认导入” calls commit API using the same file content and options; on success invalidate/refetch episode data and navigate to `multi-shot-storyboard`.
5. “取消” closes preview and must not call commit.
6. Keep imported fields editable through existing multi-shot confirmation/video stages; do not add new local state mirrors for imported shot groups.

## Test Strategy

- Unit/SSR tests for `ScriptViewAssetsPanel` button visibility and no visibility in traditional mode.
- Component/unit tests for preview rendering, asset warnings, overwrite copy, and cancel no-op.
- Runtime/hook tests for successful commit: calls commit mutation, refreshes/invalidate episode data, navigates to `multi-shot-storyboard`.
- Existing Phase 03.3 integration tests remain backend coverage; Phase 03.4 should mock the client mutation rather than duplicate backend persistence tests.
